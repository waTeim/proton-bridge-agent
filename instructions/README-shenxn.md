# Instructions for chart authoring agent (keep on‑point)

You are a creative agent who sometimes goes too deep. Stay concise and deliver a working Helm chart with minimal scope creep.

## Objective
Create a **simple Helm chart** (from a `helm create` scaffold) for **Proton Mail Bridge** using the **shenxn/protonmail-bridge** container.

## Container‑specific assumptions (shenxn/protonmail-bridge)
Source: https://github.com/shenxn/protonmail-bridge-docker

1) **Initialization is interactive** and must be done with:
   ```bash
   docker run --rm -it -v protonmail:/root shenxn/protonmail-bridge init
   ```
   In K8s, this means we need **manual first‑time setup** (kubectl exec into pod and run `init`, then `login`, then `info`, then `exit`).

2) **Runtime** is non‑interactive:
   ```bash
   docker run -d -v protonmail:/root -p 1025:25 -p 1143:143 shenxn/protonmail-bridge
   ```

3) **Ports** used:
   - SMTP: container **25** (service **1025**)
   - IMAP: container **143** (service **1143**)

4) **Persist** `/root` (or at least the bridge state under `/root`), so configuration and login survive restarts.

## Non‑negotiable chart requirements
1. **Image**: `shenxn/protonmail-bridge` (configurable in values.yaml).
2. **Ports**: expose SMTP 1025 and IMAP 1143 via Service.
3. **Persistence**: mount a PVC at `/root` (simplest, required for bridge state).
4. **No secrets in ConfigMap**. Bridge credentials are generated by `init` and stored in the PVC.
5. **NOTES.txt** must explain the manual first‑time initialization steps.

## Scope boundaries
- No ingress or public exposure by default (ClusterIP only).
- No extra sidecars/operators.
- Keep changes minimal and readable.

## Deliverables
- Updated templates (Deployment/StatefulSet, Service, PVC, NOTES.txt)
- Updated values.yaml

## Notes for the agent
- Prefer a **StatefulSet** if you want stable identity for manual setup.
- Use a single PVC mounted at `/root`.
- Document how to run `init` via `kubectl exec` and how to get the bridge’s IMAP credentials with `info`.
